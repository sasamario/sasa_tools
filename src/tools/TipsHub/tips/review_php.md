# PHP
- 継承しかしないクラスは`abstract`をつける
  - →特にカスタムRule、Request、ModelなどはBaseクラスを用意して継承するのがよい
- 継承元にコンストラクタがある場合、継承先のコンストラクタで`parent::__construct()`を実行すること
  - →継承元のコンストラクタが上書きされ呼ばれないため
- `string|null`のような型の場合、`?string`のように記載する
- `xxx|yyy|zzz`のように複数（3つ以上？）の型が想定される場合、`mixed`で指定しておきPHPDocなどで実際の型を記載する
- 自クラス内でしか使わない処理はアクセス修飾子を`private`で定義する
  - →これに限らず、適切なアクセス修飾子を指定すること
- 三項演算子で空やデフォルト値は false 側に記載すること
  - →可読性の観点から
- 変数に入れる必要がないケース（直接returnできる等）場合は変数に入れないように
  - →クセでよく変数に入れてしまいがちなので注意。ただし、説明変数としての役割がある等理由がある場合は良い
- マジックナンバーは定数ファイルなどに定義すること
  - 定義しておくことで何の値か明確になるため
- PHPのコード内ではダブルクォートではなく、シングルクォートを基本使うこと
  - ダブルクォートを使うのは変数展開するケース、エスケープシーケンス（\nなど）を使う時とすること
  - シングルクォートであれば文字列とすぐに判断できるが、ダブルクォートの場合変数などが含まれている可能性を考える必要が出るため


# Laravel
## 全体
- Carbonクラスは便利ゆえに多用してしまうため、よく使う処理はヘルパ関数として定義してそこでCarbonを呼び出すこと
- ヘルパ関数を定義する際は、必ず`function_exists()`で関数の存在チェックをすること
  - 同名の関数があった場合、意図しない結果になる可能性があるため
- 空チェックはヘルパ関数（例：`isNvl()`）で定義しておき、それを使うこと
  - 空配列、null、空文字を判別できるようなものが使いやすい
- ゼロチェックも空チェック同様、ヘルパ関数（例：isZero()）で定義しておき、それを使うこと

## ルーティング関連
- ルーティングは名前付きルーティングを使うこと
  - `route('hoge')`のように指定してリダイレクトなどする際に使う
- ルーティングでは適宜グループ化して使うこと（関連ページ、管理側等）
- ルーティングで複数のHTTP動詞に応答するルートの場合は、match()を使うこと

## View関連
- エスケープの`{{}}`は変数部分のみに対して使う方がわかりやすい。`Hello, {{ $text }}`のように
  - `{{ 'Hello' . $text }}`のようにも書けるが、文字列連結するが増えて可読性が下がるので
- ディレクティブの記述を統一すること
  - `@if()`と`@if ()`のようにスペースを入れるかどうかなどのフォーマットを統一すること
  - 公式ドキュメントには`@if ()`と書いてあるがそれを推奨しているわけではない
  - 関数的な呼び出しと捉えて、@if()のようにスペースを入れない方が良いかなと思う。なお、PHPでのifはif ()と書く
- `@php`ディレクティブは基本使わないこと。必要なデータはロジック側から渡すこと
  - viewにPHPの処理を書くことは極力避けること
- Laravelに限らずだが、`===`や`in_array()`で厳密比較する際にDB取得値をそのまま使う場合は明示的にキャストしてから行うこと
  - LaravelがというよりはPDOが例えばint型のカラムを取得する際に文字列として返すことがあるため
- checkboxでold()での選択値保持をする際は注意すること
  - checkboxは未チェック時パラメータ自体POSTされないので、old()を使っている場合第二引数を指定している場合そちらが適用されるので実装によっては注意が必要

## DB関連
- DB操作でJOINする際は、上級JOIN句を使うこと（通常のjoin()だと結合条件が複雑な場合に対応できないため）
  - [データベース：クエリビルダ 上級JOIN句](https://readouble.com/laravel/9.x/ja/queries.html)
- 結合する際は、結合先のテーブルを先に記載するようにする
- SELECT時に生のSQLを記載した場合は、`selectRaw()`よりselct()の中で`DB::raw()`を使った方がわかりやすい
  - バインド処理はこちらで行う必要があるためパラメータを渡す際は忘れずに！
- SELECT時は取得するカラムを明示的に記載すること（`select(['id', 'name'])`のように）
- LIKEを使う場合は、エスケープ用の処理をヘルパ関数などで用意しておきそれを通すこと　※Laravelに限らずだが...
  - 使用しているDBによるが、エスケープが必要な文字が含まれているケースに対応するため 
- DB操作はRepository内で完結させること
  - 例えば`DB::begintransaction()`,`DB::commit()`,`DB::rollBack()`などをServiceで呼ばず、Repositoryで呼ぶなど

## バリデーション関連
- バリデーションはカスタムルールクラス（例：ValidateE100.php）を用意して実装すること
- バリデーション用のメッセージは、`src/lang/ja/validation.php`を用意してそこに実装すること

## エラーハンドリング
- 404,419,500などLaravelがデフォルトで用意しているviewは基本出さないよう対応すること
  - 404や500に関しては、404.blade.phpといったエラーコードのviewを用意することでLaravelが自動的にそのviewを返してくれる
  - もし、これらのviewをひとまとめにしたい場合は、src/app/Exceptions/Handler.phpで対応することになる。その際はリダイレクトさせるのではなくviewをレスポンスすること（リダイレクトしてしまうと、404や500ではなく302でレスポンスされてしまうので）
  - 419に関してはリダイレクトで良い（Handler.phpでTokenMismatchExceptionの場合の処理を制御する）

## その他
- 入力、確認、完了といったフォームを実装する際は、完了画面遷移時にトークンの再生成すること（トークンを再生成することで再度リクエストされても重複登録を防ぐ）


# その他
- return前に空行を入れる。ただし、ワンライナー（一行で完結している処理）の場合は空行不要
- SQLで使う定義値（DB定義書に記載の定義値）は、VO化すると良い
  - constants.phpなどに定義する際は、VOで使う定義値はまとめておくとわかりやすい
