# TLSについて
TLS(Transport Layer Security)は、ネットワーク上で安全にデータを送受信するためのセキュリティプロトコル。
TLSはSSL(Secure Sockets Layer)の後継プロトコル。現在ではSSLは利用されておらずTLSが使われている。
ただ、慣習的に「SSL通信」や「SSL証明書」と呼ばれることがある。

## TLSが提供する3つの機能
TLSはデータを送受信するにあたりデータの安全性とセキュリティを確保するための以下3つの主要なサービスを提供している。
- 認証
  - 通信相手が正当なサーバであることを確認
- 暗号化
  - 通信内容を暗号化し、盗聴を防ぐ
- 完全性
  - 通信途中でデータの紛失、破損、改ざん、偽造がないことを保証

これらの機能を実現するために、通信開始時にTLSハンドシェイクを実施し、通信相手の認証や暗号化方式・共通鍵の決定を行う。

# TLSハンドシェイクとは
TLSハンドシェイクは、クライアントとサーバがHTTPS通信を開始する前に行うプロセス。
HTTP通信を暗号化する前に、以下の内容をお互い確認・決定する。
- 通信相手が正当なサーバであることを確認
- 利用するTLSバージョンや暗号方式を決定
- 通信に使用する共通鍵を安全に共有
- ハンドシェイクの内容が改ざんされていないことを確認

上記プロセスが完了すると、暗号化通信が行われる。
なお、TLSのv1.2とv1.3でTLSハンドシェイクの流れは異なる。現在はv1.3が主流のため、v1.3でのハンドシェイクの流れについてまとめる。

## TLS(v1.3)ハンドシェイクの流れ
```text
クライアント                            サーバ
--------------------------------------------------------

① Client Hello      ------------>

                             <------------  ② Server Hello
                                                  ③ サーバ証明書
                                                  ④ Certificate Verify
                                                  ⑤ Finished

⑥ Finished           ------------>

======== TLSハンドシェイク完了 ========

⑦ HTTPS(HTTP over TLS)通信開始
```

### 1. Client Hello(クライアント)
クライアント(ブラウザ)が最初に送るメッセージ。以下の情報を送信している。
- 対応しているTLSのバージョン
- 利用できる暗号スイート(Cipher Suites)　※安全な通信接続するために使用するアルゴリズムのセット
- クライアントランダム　※ハンドシェイクや鍵生成で利用されるランダム値
- クライアント側の鍵共有情報(Key Share)　※クライアント側で生成した公開鍵A

### 2. Server Hello(サーバ)
サーバ側はClient Helloを受信後、鍵(秘密鍵B、公開鍵Bのペア)を生成する。
この時点で、`公開鍵A と 秘密鍵B`を用いて共通鍵を生成。

サーバ側は以下の情報をクライアントに送る。
- 使用するTLSのバージョン
- 使用する暗号スイート
- サーバランダム
- サーバ側の鍵共有情報(Key Share)　※サーバ側で生成した公開鍵B

クライアント側でServer Hello受信後、`公開鍵B と 秘密鍵A`を用いて共通鍵を生成。
`公開鍵A と 秘密鍵B`を用いて生成した共通鍵と`公開鍵B と 秘密鍵A`を用いて生成した共通鍵(セッション鍵)は同じ値となる
（細かい計算は端折っているがざっくりとしたイメージはこんな感じ）。

これにより、TLSv1.2のように共通鍵をクライアント-サーバ間で送受信することなく、同じ共通鍵を導出できる。
この共通鍵暗号方式における鍵共有方法を、ECDHE法(楕円曲線Diffie-Hellman鍵共有法)という。


### 3. サーバ証明書(サーバ)
サーバ証明書をクライアントに送る。クライアントは以下を確認する。
- 信頼できる認証局(CA)が発行した証明書か
- 接続先ドメイン名と一致するか
- 有効期限内か

### 4. Certificate Verify(サーバ)
サーバ証明書の所有者であることを証明するために、秘密鍵で作った電子署名を送る。

### 5. Finished(サーバ)
Finishedメッセージを送る。
Finishedの役割は、**双方が同じ共通鍵（セッション鍵）を保持しており、ハンドシェイクの内容が改ざんされていないことを確認すること**である。具体的には、今までのハンドシェイクの内容を元に共通鍵で検証データを計算し、Finishedメッセージとして送る。

### 6. Finished(クライアント)
サーバ同様にクライアント側でもハンドシェイクの内容と共通鍵から検証データを計算し、**サーバから受信したFinishedに含まれる検証データ**と一致するか確認。

その後クライアント側も検証データを含むFinishedメッセージを送り、サーバ側で同様の検証を行う。

検証が成功すると
- 双方が同じ共通鍵(セッション鍵)を保持していること
- ハンドシェイクの内容が改ざんされていないこと
が確認され、TLSハンドシェイクが完了する。

### 7. HTTPS通信開始
以降はHTTPS通信を行う。
